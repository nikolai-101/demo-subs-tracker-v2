## 1. Контекст и проблема

У пользователя накопилось много платных сервисов (стриминг, инфраструктура, инструменты, развлечения), которые списываются регулярно. Не хватает единой картины: сколько в сумме уходит в месяц, что списывается на ближайшей неделе, какие подписки активны. Банковские выписки эту информацию дают пост-фактум и фрагментированно, агрегаторы-приложения избыточны и просят доступ к банку.

Нужен лёгкий локальный инструмент, в который пользователь сам заносит подписки и видит свою «карту трат».

## 2. Цели и не-цели

### Цели (MVP)

- Дать пользователю полный обзор активных подписок в одном экране.
- Показать **нормализованную сумму** месячных трат (годовые подписки приводятся к месячному эквиваленту: годовая / 12).
- Показать ближайшие 7 дней списаний — что и сколько спишется.
- Дать ручное подтверждение факта списания со сдвигом даты следующего платежа.

### Не-цели (вне MVP)

- Интеграция с банками / автоматический импорт транзакций.
- Аутентификация, мульти-юзер, синхронизация между устройствами.
- Уведомления (email/push).
- Мульти-валютность с автоконвертацией.
- Импорт/экспорт CSV.
- Мобильное приложение.
- Аналитика трендов и графиков сверх месячного агрегата.

## 3. Целевая аудитория

**Один пользователь** — сам автор, для личного использования. Никакого логина, регистрации, ролей. Один локальный экземпляр приложения = один пользователь.

## 4. Сценарии использования

### S1. Завести новую подписку
Пользователь нажимает «Добавить подписку», вводит название, сумму, валюту, период (месяц/год), дату следующего списания, опционально категорию. Подписка появляется в списке и учитывается в агрегатах дашборда.

### S2. Посмотреть месячные траты
Открывает главный экран → видит крупную цифру «X в месяц» (нормализованную сумму всех активных подписок) и список «Ближайшие 7 дней» с конкретными датами и суммами.

### S3. Подтвердить факт списания
Получает реальное уведомление от банка → открывает приложение → находит подписку → нажимает «Списалось». Дата следующего списания сдвигается на +1 месяц или +1 год (в зависимости от периода). В истории платежей появляется запись.

### S4. Отредактировать подписку
Цена изменилась / поменялся период → редактирует запись. Изменения применяются к будущим расчётам, история не переписывается.

### S5. Отменить подписку
Сервис больше не нужен → помечает «отменена». Подписка перестаёт учитываться в активных агрегатах, но остаётся в архиве для истории.

### S6. Сгруппировать по категориям
Хочет понять «сколько уходит на инфраструктуру vs развлечения» → фильтрует/сортирует список подписок по категории.

## 5. Функциональные требования

### 5.1. Подписка (Subscription)

Поля:

| Поле | Тип | Обязательность | Примечания |
|---|---|---|---|
| `id` | uuid/int | да | PK |
| `name` | string | да | «Netflix», «AWS», «GitHub Pro» |
| `amount` | decimal | да | Сумма списания за период |
| `currency` | string (ISO 4217) | да | Например `RUB`, `USD`. В MVP — пользователь сам выбирает одну, конвертации нет. |
| `period` | enum (`monthly` \| `yearly`) | да | |
| `next_charge_date` | date | да | Дата следующего ожидаемого списания |
| `category_id` | FK → categories | нет | |
| `note` | string | нет | Свободный комментарий |
| `status` | enum (`active` \| `cancelled`) | да, default `active` | |
| `created_at` | timestamp | да | |
| `updated_at` | timestamp | да | |

### 5.2. Категория (Category)

| Поле | Тип | Обязательность |
|---|---|---|
| `id` | int | да (PK) |
| `name` | string | да, unique |
| `color` | string (hex) | нет |

Категории создаются и редактируются отдельно (страница «Категории» или модал из формы подписки).

### 5.3. История платежей (PaymentEvent)

Каждое подтверждение «списалось» создаёт запись:

| Поле | Тип |
|---|---|
| `id` | int |
| `subscription_id` | FK |
| `amount` | decimal (снэпшот цены на момент списания) |
| `currency` | string |
| `charged_at` | date |
| `created_at` | timestamp |

Используется для аудита и потенциальной будущей аналитики (вне MVP).

### 5.4. Главный экран (Dashboard)

Отображает:

1. **Карточка «В месяц»** — нормализованная сумма всех активных подписок:
   - Для `monthly` → `amount`
   - Для `yearly` → `amount / 12`
   - Сумма по всем валютам отображается отдельными строками (если есть и `USD` и `RUB`, показываются две строки). Конвертация не делается.
2. **Список «Ближайшие 7 дней»** — все активные подписки, у которых `next_charge_date` попадает в окно `[today, today + 7]`. Сортировка по возрастанию даты. Для каждой: название, сумма, валюта, дата (с относительным форматом «завтра», «через 3 дня»).
3. **Бейдж «Просрочено»** — если у активной подписки `next_charge_date < today`. Подсветить и предложить кнопку «Списалось».

### 5.5. Список подписок

- Таблица/карточки всех подписок (с фильтром: активные/отменённые/все).
- Фильтр по категории.
- Сортировка по: дате следующего списания, сумме, названию.
- Действия по строке: «Списалось», «Редактировать», «Отменить» (для активных), «Восстановить» (для отменённых), «Удалить».

### 5.6. Действие «Списалось» (manual confirm)

При нажатии:
1. Создать запись `PaymentEvent` со снэпшотом текущей цены, валюты и сегодняшней даты.
2. Сдвинуть `next_charge_date`:
   - `monthly` → `+1 месяц` (использовать корректную календарную арифметику: 31 января → 28/29 февраля; день месяца сохраняется, если возможно).
   - `yearly` → `+1 год`.
3. Если подписка была «просрочена» (несколько периодов пропущено), сдвигать **на один период за нажатие** — пользователь явно подтверждает каждое списание отдельно. Альтернатива (батч-подтверждение нескольких пропущенных периодов) — вне MVP.

## 6. Нефункциональные требования

- **Локальный запуск:** `npm run dev`, открывается на `localhost:3000`. SQLite-файл лежит в корне проекта (например `./data/subscriptions.db`), `.gitignore` исключает его.
- **Производительность:** ожидаемый объём — десятки, максимум сотни подписок. Без оптимизаций.
- **Бэкап:** пользователь сам копирует `subscriptions.db` (документировать в README). Автобэкап — вне MVP.
- **Доступность:** не критична, инструмент локальный.
- **i18n:** только русский в MVP.

## 7. Технический стек

- **Фреймворк:** Next.js (App Router, последняя стабильная версия).
- **БД:** SQLite через [better-sqlite3](https://github.com/WiseLibs/better-sqlite3) (синхронный, простой, идеален для локального single-user). Альтернатива: Drizzle ORM поверх better-sqlite3 — рекомендуется, если хочется типобезопасные миграции; иначе чистый SQL с миграциями вручную.
- **UI:** Tailwind CSS.
- **Формы/валидация:** react-hook-form + zod.
- **Даты:** [date-fns](https://date-fns.org/) (для арифметики месяцев/лет с корректной обработкой граничных дат).
- **Server logic:** Server Actions + API routes по необходимости (CRUD через Server Actions).
- **Тесты:** не требуются в MVP, но для логики сдвига дат и нормализации сумм желательно покрыть юнит-тестами (Vitest).

## 8. Модель данных (DDL)

```sql
CREATE TABLE categories (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL UNIQUE,
  color TEXT
);

CREATE TABLE subscriptions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  amount REAL NOT NULL,
  currency TEXT NOT NULL,
  period TEXT NOT NULL CHECK (period IN ('monthly','yearly')),
  next_charge_date TEXT NOT NULL,         -- ISO date YYYY-MM-DD
  category_id INTEGER REFERENCES categories(id) ON DELETE SET NULL,
  note TEXT,
  status TEXT NOT NULL DEFAULT 'active'
    CHECK (status IN ('active','cancelled')),
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_subscriptions_next_charge
  ON subscriptions(next_charge_date)
  WHERE status = 'active';

CREATE TABLE payment_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  subscription_id INTEGER NOT NULL
    REFERENCES subscriptions(id) ON DELETE CASCADE,
  amount REAL NOT NULL,
  currency TEXT NOT NULL,
  charged_at TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_events_subscription ON payment_events(subscription_id);
```

## 9. План разработки (фазы)

1. **Каркас.** Next.js + Tailwind + shadcn/ui + better-sqlite3, миграция schema, db.ts singleton.
2. **CRUD подписок** без категорий: форма (react-hook-form + zod), список, редактирование, удаление.
3. **Категории.** CRUD и привязка к подпискам.
4. **Дашборд.** Нормализованная сумма + блок «Ближайшие 7 дней» + бейдж «Просрочено».
5. **«Списалось».** Создание `PaymentEvent` + сдвиг даты с покрытием юнит-тестами.
6. **Полировка UX.** Пустые состояния, подтверждения удаления, аккуратная типографика на дашборде.

## 10. Открытые вопросы

- **Удаление vs отмена.** В UI должны быть оба действия или только «отменить» (с возможностью полного удаления только из архива)? Рекомендация: только «отменить» в основном потоке; «удалить» — действие из архива, чтобы не терять историю случайно.
- **Базовая валюта.** В MVP — несколько строк по валютам. Если в реальности подписки только в одной валюте — можно вообще не показывать строку с валютой, упростив UI. Решить после первого занесения данных.
- **Календарная арифметика.** 31 января + 1 месяц = 28 февраля (или 29 в високосном)? Использовать поведение `date-fns/addMonths` (cap-to-last-day), зафиксировать решение в коде с тестами.
- **Миграции БД.** Достаточно ли простого «прогнать schema.sql при старте, если таблиц нет», или нужен механизм версионирования с самого начала? Для MVP — достаточно простого; ввести версионирование при первом изменении схемы.

## 11. Метрики успеха

Это персональный инструмент, поэтому метрики качественные:

- Пользователь занёс все свои текущие подписки (>20) без раздражения от UX.
- На дашборд можно посмотреть и за 3 секунды понять, сколько уходит в месяц и что списывается на этой неделе.
- За месяц использования подтверждение «списалось» не превращается в тяжёлую рутину (если превращается — нужен следующий шаг: автопродление с post-факто корректировкой).
